# Coding Conventions

**Analysis Date:** 2026-04-25

## Naming Patterns

**Files:**

- TypeScript modules: `camelCase.ts` â€” e.g. `src/lib/server/api/utils/resolveConversation.ts`, `src/lib/server/logger.ts`, `src/lib/utils/marked.ts`.
- Svelte components: `PascalCase.svelte` â€” e.g. `src/lib/components/chat/ChatMessage.svelte`, `src/lib/components/chat/MarkdownRenderer.svelte`, `src/lib/components/chat/ChatModelPicker.svelte`.
- Test files mirror their subject and use `*.spec.ts` for server/util suites and `*.svelte.test.ts` for component browser suites â€” e.g. `marked.ts` -> `marked.spec.ts`, `MarkdownRenderer.svelte` -> `MarkdownRenderer.svelte.test.ts`.
- SvelteKit endpoints: `+server.ts` (route handlers), `+page.svelte`, `+page.server.ts`, `+layout.svelte`. Dynamic segments use `[id]` brackets: `src/routes/api/v2/conversations/[id]/+server.ts`.
- Test helpers and shared test data live in `__tests__/` directories at the appropriate level â€” e.g. `src/lib/server/api/__tests__/testHelpers.ts`.

**Functions:**

- `camelCase` for ordinary functions and methods. Examples: `resolveConversation`, `requireAuth`, `superjsonResponse`, `pickToolsCapableModel`, `sanitizeGeneratedTitle`, `createTestUser`, `cleanupTestData`.
- `PascalCase` only for classes (e.g. `AbortRegistry`) and Svelte components.
- Boolean predicates use `is*` / `has*` / `can*`: `isMessageToolUpdate`, `isAssistantGenerationTerminal`, `isConversationGenerationActive`, `canUseHermesTools`.
- Factories begin with `create*`: `createTestUser`, `createTestConversation`, `createTestLocals`, `createSettingsStore`, `createCookiesMock`.
- Hook-like store accessors begin with `use*`: `useSettingsStore` (in `src/lib/stores/settings.ts`), `usePublicConfig`.

**Variables:**

- Local variables and parameters: `camelCase`. Constants for env-derived data are `SCREAMING_SNAKE_CASE` and pulled from typed config (`config.LLM_ROUTER_HERMES_MODEL`, `CONV_NUM_PER_PAGE`).
- Private/unused parameters prefixed with `_` so the lint rule ignores them â€” `argsIgnorePattern: "^_"` in `.eslintrc.cjs`. Example in `src/lib/components/chat/ChatMessage.svelte`: `let { isAuthor: _isAuthor = true, readOnly: _readOnly = false, ... } = $props();`.

**Types:**

- Type aliases and interfaces: `PascalCase` â€” `Conversation`, `Message`, `User`, `Model`, `Settings`, `App.Locals`, `RequestHandler`, `RouterFollowUp`.
- Component prop interfaces are local and named `Props`: `interface Props { ... }` followed by `let { ... }: Props = $props();` (see `ChatModelPicker.svelte`, `ChatWindow.svelte`, `ChatMessage.svelte`).

## Code Style

**Formatting:** Prettier (`prettier@^3.5.3`) via `.prettierrc`:

- `"useTabs": true` â€” tabs, not spaces.
- `"trailingComma": "es5"`.
- `"printWidth": 100`.
- Plugins: `prettier-plugin-svelte` and `prettier-plugin-tailwindcss` (auto-sorts Tailwind classes).
- Run `npm run format` to apply, `npm run lint` to verify (which runs `prettier --check . && eslint .`).

**Linting:** ESLint (`eslint@^8.28.0`) via `.eslintrc.cjs`. Extends `eslint:recommended`, `plugin:@typescript-eslint/recommended`, `plugin:svelte/recommended`, `prettier`. Hard rules:

- `@typescript-eslint/no-explicit-any: error` â€” never use `any`. Prefer `unknown` and narrow, or `as never` only when forced (see test files passing `{ locals, url } as never` to `RequestHandler`s).
- `@typescript-eslint/no-non-null-assertion: error` â€” never use `!`. Use optional chaining and explicit guards instead (see `locals.user?._id` in `requireAuth.ts`).
- `@typescript-eslint/no-unused-vars: error` with `argsIgnorePattern: "^_"`.
- `object-shorthand: ["error", "always"]`.
- `no-empty: off`, `require-yield: off`.

**TypeScript:** `tsconfig.json` extends SvelteKit's generated config and turns on `strict`, `allowJs`, `checkJs`, `esModuleInterop`, `forceConsistentCasingInFileNames`, `resolveJsonModule`, `target: ES2018`. Strict null checks are in effect â€” code uses optional chaining heavily.

## Import Organization

**Order (observed in `src/lib/components/chat/ChatWindow.svelte`, `src/routes/api/v2/conversations/+server.ts`):**

1. SvelteKit framework imports (`@sveltejs/kit`, `svelte`, `svelte/store`, `svelte/transition`).
2. SvelteKit module aliases (`$app/environment`, `$app/navigation`, `$app/paths`, `$app/state`).
3. Project alias imports (`$lib/...`).
4. Third-party packages (`@huggingface/inference`, `mongodb`, `superjson`, `uuid`, `zod`).
5. Icon imports last (`~icons/carbon/*`, `~icons/lucide/*`).
6. Relative imports (`./Foo.svelte`, `../Bar.svelte`).

**Path Aliases:**

- `$lib` -> `src/lib` (used everywhere: `$lib/server/database`, `$lib/types/Message`, `$lib/stores/settings`, `$lib/utils/...`).
- `$app/*` â€” SvelteKit built-ins (`$app/environment`, `$app/navigation`, `$app/paths`, `$app/state`).
- `$env/dynamic/private`, `$env/dynamic/public` â€” SvelteKit env (mocked in test setup).
- `~icons/carbon/*` and `~icons/lucide/*` â€” auto-generated by `unplugin-icons` Vite plugin.

Tests prefer the alias form when reaching outside `__tests__/` (e.g. `import { collections } from "$lib/server/database"`) and relative paths when reaching local helpers (`import { cleanupTestData } from "./testHelpers"`).

## Error Handling

**Server endpoints throw via SvelteKit's `error()` helper** rather than returning custom error objects. The helper sets HTTP status and is caught by SvelteKit:

```ts
// src/lib/server/api/utils/requireAuth.ts
import { error } from "@sveltejs/kit";

export function requireAuth(locals: App.Locals): void {
	if (!locals.user?._id && !locals.sessionId) {
		error(401, "Must have a valid session or user");
	}
}
```

```ts
// src/lib/server/api/utils/resolveConversation.ts
if (!conversation) {
	error(404, "Conversation not found.");
}
```

**Try/catch is used to translate non-HTTP errors into HTTP errors and to re-throw existing HTTP errors unchanged:**

```ts
// src/lib/server/api/utils/resolveModel.ts
try {
	const { models } = await import("$lib/server/models");
	const found = models.find((m) => m.id === modelId);
	if (!found || found.unlisted) error(404, "Model not found");
	return found;
} catch (e) {
	if (e && typeof e === "object" && "status" in e) throw e; // pass HttpError through
	error(500, "Models not available");
}
```

**Validation errors use `error(400, "...")`** before any database work (`ObjectId.isValid(id)` checks, Zod schemas like `validModelIdSchema.safeParse(model)`).

**Body responses use `superjsonResponse(data)`** so Date/ObjectId/etc. survive the wire round-trip. Never `return new Response(JSON.stringify(...))` directly from API endpoints in `src/routes/api/v2/`.

## Logging

**Framework:** `pino` via `src/lib/server/logger.ts`. Use `import { logger } from "$lib/server/logger"`.

- In `dev` mode, transports through `pino-pretty` for colorized console output.
- Level controlled by `config.LOG_LEVEL`, defaults to `"info"`.
- `mixin()` automatically enriches every log line with `request_id`, `url`, `ip`, `user`, `status_code` from `getRequestContext()` (request-scoped).

**Patterns:** Log structured data, not concatenated strings: `logger.warn({ modelId }, "model not found")` rather than `logger.warn("model " + id + " not found")`. Browser code uses plain `console.*`; never import the server logger from a `+page.svelte` or `src/lib/components/`.

## Comments

**When to Comment:** Comments document non-obvious behavior, business rules, and the _why_ â€” not the _what_. Examples:

- `src/lib/server/api/utils/resolveConversation.ts` documents shared-link semantics (`// shared link of length 7`).
- `src/routes/api/v2/conversations/+server.ts` notes legacy fields kept for iOS clients (`id: conv._id, // legacy param iOS`).
- `src/lib/utils/tree/treeHelpers.spec.ts` uses ASCII tree diagrams to explain fixture shapes.

**JSDoc/TSDoc:** Used on exported helpers in `src/lib/server/api/utils/` to describe contract and error behavior:

```ts
/**
 * Throws 401 if neither a user._id nor sessionId is present in locals.
 */
export function requireAuth(locals: App.Locals): void { ... }
```

Avoid `// TODO` without an owner; prefer issues. Avoid restating types â€” TypeScript already documents shape.

## Function Design

**Size:** Small, single-purpose functions. `requireAuth` (4 lines), `superjsonResponse` (10 lines), `resolveModel` (~25 lines). `resolveConversation` reaches ~70 lines because it inlines two related lookup paths; in practice modules stay well under 200 lines.

**Parameters:**

- 0â€“3 positional params is the norm (`requireAuth(locals)`, `resolveConversation(id, locals, fromShare?)`).
- Once you exceed 3 params or need optional fields, accept an options object: `upsertEntitlementFromStripeSubscription({ userId, stripeCustomerId, stripeSubscriptionId, ... })`.
- Never spread untyped objects â€” destructure with explicit type annotations on the destructure target.

**Return Values:**

- Endpoint handlers return `Response`. Internal services return plain TS objects/promises.
- Avoid throwing for control flow except via `error(...)` (which is the SvelteKit-blessed mechanism for HTTP error short-circuits).
- Prefer narrow return types â€” projection-typed Mongo finds (`.project<Pick<Conversation, "_id" | "title" | ...>>(...)` in `routes/api/v2/conversations/+server.ts`).

## Module Design

**Exports:** Named exports only. Default exports are reserved for Svelte components and Vite-generated artifacts. Most modules export 1â€“3 functions plus their types.

**Barrel Files:** Not used. Import each helper from its own file: `import { requireAuth } from "$lib/server/api/utils/requireAuth"`. This keeps tree-shaking and SSR boundary checks unambiguous.

**Server vs client boundary:** Anything imported by a client component must be safe in the browser. Server-only code lives in `src/lib/server/**` and SvelteKit enforces it cannot be imported from non-`+server.ts` / non-`.server.ts` files. Tests for server code are run in the `node` workspace.

## Svelte 5 Conventions

**Runes:** All component state uses Svelte 5 runes â€” `$state`, `$derived`, `$effect`, `$props`, `$bindable`. Legacy `export let` is not used.

```svelte
<!-- src/lib/components/chat/ChatModelPicker.svelte -->
let rootEl: HTMLDivElement | undefined = $state(); let isOpen = $state(false); let modelFilter = $state("");
let queryTokens = $derived(normalize(modelFilter).trim().split(/\s+/).filter(Boolean));
```

**Component Props:** Declared via a local `interface Props { ... }`, then a single `$props()` destructure with explicit types and defaults:

```svelte
<!-- src/lib/components/chat/ChatModelPicker.svelte -->
interface Props {
	models: Model[];
	currentModel: Model;
	disabled?: boolean;
}
let { models, currentModel, disabled = false }: Props = $props();
```

Two-way bindings use `$bindable()`:

```svelte
<!-- src/lib/components/chat/ChatWindow.svelte -->
let {
	files = $bindable([]),
	draft = $bindable(""),
	...
}: Props = $props();
```

When a prop is intentionally unused but kept for API compatibility, alias it with a leading underscore: `isAuthor: _isAuthor = true` (see `ChatMessage.svelte`).

**Stores:** Live in `src/lib/stores/`. Two patterns coexist:

- **Plain writable stores** â€” `errors.ts`, `loading.ts`, `pendingMessage.ts`, `shareModal.ts` etc. exported as singletons via `writable(...)`.
- **Context-scoped factories** â€” for stores that need per-page state (settings, MCP servers), the module exports `createXStore()` that calls `setContext(...)` and a `useXStore()` accessor that calls `getContext(...)`. See `src/lib/stores/settings.ts`:

```ts
export function useSettingsStore() {
	return getContext<SettingsStoreWritable>("settings");
}
export function createSettingsStore(initialValue: Omit<SettingsStore, "recentlySaved">) { ... }
```

Reactive state local to a single component should use `$state` rather than a store; reach for a store only when the value crosses component boundaries.

**Side Effects:** Use `$effect(...)` for DOM and async side effects. Use `tick()` from `svelte` when you need to read the DOM after a state change in the same task.
